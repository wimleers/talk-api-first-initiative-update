Vision & goals

An unparallelled API-first platform

All data available:

  • RESTfully, in any format (JSON, HAL+JSON, JSON API)
  • non-RESTfully
    • via GraphQL
    • via CouchDB protocol, for replication & syncing

… using any authentication mechanism (Cookie, Basic Auth, OAuth2)

Must have
  • Make D8 REST usable
  • Make D8 REST best-in-class
Should have
  • JSON API support (targeting 8.3)
  • UUID and revision support (targeting 8.3)
  • GraphQL support (targeting post-8.3)
Could have
  • CouchDB support
  • OAuth2 support
REST API best in class
Drupal 8’s REST API should be usable by 8.2 and best-in-class by 8.3, including features such as configuration entity support, translation and file upload support, and pagination.
JSON API core experimental
JSON API is highly opinionated and prescriptive, which means better interoperability and less bikeshedding about details. A possible replacement for HAL+JSON.
GraphQL contrib stable
GraphQL is well-suited for fetching data to be used for UI components and makes data available in a non-RESTful manner.
UUIDs and revisions
D8 REST lacks UUID support.
The “Entity” REST resource does not support revisions and uses IDs; RELAXed Web Services adds support for both.
OAuth 2 Bearer Tokens
REST uses Basic/Cookie Auth.
Basic Auth is limited and insecure over HTTP connections, and Cookie Auth is only useful for browsers (i.e. progressive decoupling). A more secure auth mechanism is needed.
replication.io CouchDB support
REST lacks content replicability.
RELAXed Web Services makes it possible to use REST for complex workflows or content replication for deployment purposes or for offline-first implementations using PouchDB.
